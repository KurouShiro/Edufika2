## Sistem Session Token Baru (Server Authoritative)

Session token bukan JWT login biasa.

Session token adalah:

* Token acak yang dibuat proktor
* Sekali pakai untuk "mengklaim sesi"
* Setelah itu diganti dengan Access Signature berumur pendek

### Siklus Hidup Token

Token Sesi Ujian
- Dibuat oleh Proktor
- Pair token dibuat per sesi (`S-*` siswa, `A-*` admin/proktor) dengan dukungan multi-student token dalam satu sesi
- Batch token dapat dibuat sekaligus (contoh 300 siswa + 1 admin) dari satu sesi backend
- Di-claim oleh Device Siswa
- Device Binding Dilakukan
- Kebijakan PIN proktor diturunkan ke binding siswa saat claim
- Timestamp kedaluwarsa dikirim ke klien
- Access Signature Rotasi Setiap sekitar 5 Menit
- Heartbeat wajib setiap 5-10 detik
- Risiko dihitung server
- Proktor dapat pause/resume session secara live
- Proktor dapat reissue signature siswa tanpa membuat token baru
- Server dapat mengakhiri sesi kapan saja

## Arsitektur Backend (Node.js / Express API)

Backend berfungsi sebagai:
**Otoritas Sesi Ujian (Satu Sumber Kebenaran)**

Bukan sekadar server autentikasi.

### Tanggung Jawab Backend

* Membuat & mengelola exam session
* Validasi heartbeat perangkat
* Menghitung risk score
* Rotasi access signature
* Runtime database menggunakan MySQL/MariaDB (`mysql2`) dengan migration tracker `schema_migrations`
* Menjadi sumber kebenaran whitelist URL (server authoritative)
* Mengelola launch URL untuk mode lockdown web browser (Google Forms/web host)
* Mengelola kebijakan Proctor PIN harian
* Proctor PIN policy bersifat **session-scoped**: template di `session_student_pin_templates`, lalu dibinding ke `binding_id` siswa di `session_proctor_pins`
* Mengelola token per exam session: multi `S-*` (student) dalam satu sesi + satu `A-*` (admin/proctor)
* Mendukung bulk set PIN untuk seluruh token siswa sesi yang sama (`/session/proctor-pin/set-all`)
* Mengirim metadata kedaluwarsa token ke klien untuk countdown/enforcement
* Mendukung reconnect signature (`/session/reconnect`) untuk pemulihan signature/kontrol admin tanpa membuat sesi baru
* Menyediakan kontrol sesi live dari proktor: pause/resume/reissue signature/revoke student
* Menyediakan endpoint monitor token realtime (status online/offline/revoked/expired + IP/perangkat)
* Status monitor backend memprioritaskan lock/revoke state agar token revoke tidak tertutup oleh status expired
* Timeout heartbeat mengunci binding siswa yang stale tanpa mematikan seluruh sesi
* Fokus pada session authority + launch URL host pihak ketiga (tanpa menyimpan bank soal lokal)
* Mencatat seluruh aktivitas audit
* Auto-archive + cleanup sesi selesai/revoke ke `session_cleanup_audit`
* Live revoke session oleh proktor
* Siap dikontainerisasi untuk deployment (`Dockerfile` + `docker-compose.yml`)

## Database Schema & Flow (MySQL/MariaDB)

### Tabel Inti (Session Authority)

* `exam_sessions` (metadata sesi + status + `exam_mode` + waktu start/end)
* `session_tokens` (token per sesi: admin + banyak siswa, role, claimed_at, expires_at)
* `device_bindings` (binding token->device, fingerprint, IP, device_name, signature_version, risk_score, lock, last_seen)
* `heartbeats` (telemetry periodik per binding)
* `violations` (event pelanggaran per binding)
* `session_whitelist` (URL whitelist per sesi)
* `session_browser_targets` (launch URL + provider + lock_to_host per sesi)
* `session_proctor_pins` (PIN proktor per binding, efektif per hari)
* `session_student_pin_templates` (template PIN proktor per student token, efektif per hari)
* `session_cleanup_audit` (arsip sesi selesai/revoke)

### Alur Admin (DB)

* Proktor membuat sesi -> `exam_sessions`
* Generate token admin + siswa -> `session_tokens`
* Admin claim token -> `device_bindings` (role admin)
* Set launch URL -> `session_browser_targets`
* Set whitelist -> `session_whitelist`
* Set Proctor PIN per token -> `session_student_pin_templates`
* Apply PIN per binding siswa -> `session_proctor_pins`
* Monitoring -> `device_bindings` + `heartbeats` + `violations`

### Alur Siswa (DB)

* Siswa claim token -> update `session_tokens.claimed` + `device_bindings`
* Heartbeat -> `heartbeats` + update `device_bindings.last_seen_at`
* Violation -> `violations`
* Whitelist/launch -> `session_whitelist` + `session_browser_targets`

### Alur In-App Quiz (Mode Ujian)

* `exam_sessions.exam_mode` menentukan mode (BROWSER_LOCKDOWN/HYBRID/IN_APP_QUIZ)
* Konfigurasi kuis -> `quiz_exams`
* Subject kuis -> `quiz_subjects`
* Soal kuis -> `quiz_questions` + `quiz_question_options`
* Assign token siswa -> `quiz_token_assignments`
* Attempt siswa -> `quiz_student_attempts` (termasuk profil siswa)
* Jawaban siswa -> `quiz_student_answers`
* Audit aksi proktor/guru -> `quiz_teacher_submissions`

### Pengakhiran Sesi & Pembersihan (DB)

* Sesi dianggap berakhir hanya saat status `FINISHED` atau `REVOKED` + `end_time` terisi.
* Pemicu selesai: endpoint `/session/finish` (FINISHED) atau `/admin/revoke` (REVOKED).
* Logout admin hanya mengakhiri state lokal (UI/klien), **tidak** mengakhiri sesi di backend.
* Setelah grace period, sesi berakhir di-archive ke `session_cleanup_audit` lalu data sesi dihapus (cascade).

POST /session/create -> Proktor membuat sesi
POST /session/claim -> Siswa masuk ujian
POST /session/heartbeat -> Validasi liveness
POST /session/reconnect -> Refresh signature & pulihkan state binding/sesi

POST /session/finish -> Akhiri ujian

GET /session/whitelist -> Ambil whitelist aktif dari server
POST /session/whitelist/add -> Tambah URL whitelist (server authoritative)
POST /session/whitelist/verify -> Validasi URL terhadap whitelist server
GET /exam/launch -> Ambil launch URL browser exam

POST /session/proctor-pin/set -> Simpan PIN proktor (refresh harian)

POST /session/proctor-pin/verify -> Verifikasi PIN proktor saat exit exam
GET /session/proctor-pin/status -> Cek status PIN proktor hari ini
GET /health -> Tes koneksi backend dari klien (DeveloperAccess/Admin)

## Heartbeat Protocol (Wajib)

Aplikasi mengirim status:

{
focus: true,
multi_window: false,
network_state: stable,
device_state: normal,
heartbeat_seq: n,
timestamp: xxx
}

Jika heartbeat berhenti >30 detik:
Binding siswa dapat masuk status DEGRADED/SUSPENDED; pada batas lock hanya binding stale yang dikunci (bukan seluruh sesi).
Jika proktor mem-pause sesi:
heartbeat tetap tercatat tetapi status sesi `PAUSED` sampai proktor resume.

## Mesin Penilaian Risiko (Sisi Server)

Setiap kejadian meningkatkan skor:

| Kejadian             | Skor  |
| -------------------- | ----- |
| App di-background    | +3    |
| Overlay terdeteksi   | +5    |
| Accessibility aktif  | +5    |
| Koneksi terputus     | +2    |
| Pelanggaran berulang | +6    |

Jika melewati threshold:
Session -> LOCKED (butuh verifikasi proktor).

Selalu diverifikasi via API.

Navigasi keluar domain langsung dianggap pelanggaran.

## Mode Kiosk (Soft Enforcement Saja)

Menggunakan:

* Immersive mode
* Focus monitoring
* Behavioral detection
* Kiosk mode aktif default pada startup app (state reset tiap launch)
* RN host mendeteksi split-screen dan memicu lock/violation flow otomatis
* Reset state setiap launch (kiosk disable state tidak dipersisten antar reboot app)
* Setelah logout, soft-kiosk dilepas agar user dapat keluar aplikasi secara normal

Tidak menggunakan Device Owner / MDM.
Model berbasis observasi dan audit, bukan kontrol penuh OS.
